# บทที่ 1 บทนำ

## 1.1 ความเป็นมาและความสำคัญของปัญหา

การให้บริการข้อมูลหลักสูตรในระดับบัณฑิตศึกษาของมหาวิทยาลัยไทยประสบกับความท้าทายหลายประการที่ส่งผลต่อทั้งประสบการณ์ของนิสิตและภาระงานของเจ้าหน้าที่หลักสูตร เอกสารที่เกี่ยวข้องกับการเรียน เช่น คู่มือนักศึกษา ระเบียบข้อบังคับ ปฏิทินการศึกษา แบบฟอร์มการสอบ และประกาศทุนการศึกษา ถูกจัดเก็บในรูปแบบไฟล์ดิจิทัลหลากหลายประเภท ทั้ง PDF, Word, Excel และหน้าเว็บไซต์ ปริมาณเอกสารที่มากและความซับซ้อนของเนื้อหาทำให้เกิดภาวะข้อมูลล้นเกิน (Information Overload) ซึ่งเป็นอุปสรรคต่อการที่นิสิตจะค้นหาคำตอบด้วยตนเองได้อย่างมีประสิทธิภาพ

ผลลัพธ์ที่ตามมาคือ นิสิตจำนวนมากเลือกที่จะสอบถามเจ้าหน้าที่หลักสูตรโดยตรงแทนการอ่านเอกสาร การสำรวจเบื้องต้นในขั้นตอนการสัมภาษณ์เชิงลึกของการศึกษานี้พบว่าเจ้าหน้าที่หลักสูตรต้องใช้เวลาระหว่าง 1 ถึง 5 ชั่วโมงต่อวัน หรือคิดเป็นสัดส่วนประมาณ 50-70 เปอร์เซ็นต์ของเวลาทำงาน ไปกับการตอบคำถามที่ซ้ำกันเป็นระยะๆ เช่น กำหนดการสอบ ขั้นตอนการเตรียมเอกสารสอบโครงร่างวิทยานิพนธ์ หลักเกณฑ์การสำเร็จการศึกษา และการขอทุนการศึกษา ภาระงานในส่วนนี้นอกจากจะทำให้เจ้าหน้าที่ขาดเวลาสำหรับงานบริหารและงานเอกสารที่ซับซ้อนแล้ว ยังส่งผลให้นิสิตต้องรอคำตอบเฉลี่ย 1 ถึง 3 วันทำการ ซึ่งในกรณีที่ข้อมูลมีกำหนดเวลาที่เร่งด่วน เช่น การลงทะเบียนเรียน การชำระค่าเล่าเรียน หรือการยื่นเอกสารสอบจบ ความล่าช้านี้อาจก่อให้เกิดผลกระทบต่อการศึกษาของนิสิต

เทคโนโลยี Generative AI ที่พัฒนาอย่างก้าวกระโดดในช่วงปี 2565-2569 เปิดโอกาสให้สามารถออกแบบระบบตอบคำถามอัตโนมัติที่มีคุณภาพสูงขึ้นอย่างมีนัยสำคัญ โดยเฉพาะอย่างยิ่งการผสานเทคนิค Retrieval-Augmented Generation (RAG) ที่นำเสนอโดย {Lewis, 2020 #1} ซึ่งเป็นแนวทางที่ผสมผสานความสามารถของโมเดลภาษาขนาดใหญ่ในการสร้างคำตอบที่เป็นธรรมชาติ เข้ากับระบบสืบค้นข้อมูลที่ดึงเนื้อหามาจากเอกสารต้นฉบับเพื่อลดปัญหาการสร้างข้อมูลเท็จ (Hallucination) ที่เป็นข้อจำกัดของโมเดลภาษาขนาดใหญ่แบบไม่มีการอ้างอิง การปรับใช้ RAG ในบริบทของหลักสูตรการศึกษาไทยจึงเป็นโอกาสในการแก้ปัญหาข้อมูลล้นเกินและภาระงานตอบคำถามซ้ำไปพร้อมกัน

การเลือกแพลตฟอร์ม LINE Official Account เป็นช่องทางหลักของระบบสะท้อนพฤติกรรมการสื่อสารของนิสิตไทยที่ใช้ LINE เป็นช่องทางหลักในการติดต่อในชีวิตประจำวัน การออกแบบระบบบนแพลตฟอร์มที่ผู้ใช้คุ้นเคยอยู่แล้วช่วยลดอุปสรรคในการยอมรับเทคโนโลยี (Adoption Barrier) และช่วยให้นิสิตสามารถเข้าถึงข้อมูลได้ทันทีตลอด 24 ชั่วโมงผ่านอุปกรณ์ที่พกติดตัวเสมอ

นอกเหนือจากการแก้ปัญหาเฉพาะของหลักสูตรเดียวแล้ว การออกแบบระบบในสถาปัตยกรรมแบบ Multi-tenant Software-as-a-Service ตามแนวคิดของ {Bezemer, 2010 #3} ยังเปิดโอกาสในการขยายผลการใช้งานไปยังหลักสูตรหรือคณะอื่นที่มีลักษณะปัญหาใกล้เคียงกัน การสามารถรองรับผู้ใช้งานหลายองค์กรภายใต้ระบบเดียวกันทำให้ต้นทุนคงที่ถูกกระจายไปยังผู้ใช้งานจำนวนมาก และสามารถพัฒนาโมเดลธุรกิจ B2B SaaS ที่ยั่งยืนได้ในระยะยาว

## 1.2 คำถามวิจัย

การศึกษานี้มีคำถามวิจัยหลักสี่ข้อที่ครอบคลุมมิติของคุณภาพระบบ การยอมรับของผู้ใช้งาน และความเป็นไปได้เชิงพาณิชย์ ดังนี้

1. ระบบ RAG Chatbot ที่ดำเนินการบนแพลตฟอร์ม LINE Official Account สามารถตอบคำถามของนิสิตที่เกี่ยวข้องกับหลักสูตรได้ด้วยคุณภาพและความถูกต้องในระดับที่เพียงพอสำหรับการนำไปใช้งานจริงเพียงใด
2. เจ้าหน้าที่หลักสูตรสามารถใช้ระบบจัดการหลังบ้าน (Admin Portal) ในการเพิ่มเอกสารใหม่เข้าสู่ฐานความรู้ ตรวจสอบบันทึกการสนทนา และติดตามสถิติการใช้งาน ได้อย่างสะดวกเพียงใดโดยไม่ต้องการผู้เชี่ยวชาญด้านเทคนิคประกอบการใช้งาน
3. นิสิตผู้ใช้งานมีระดับการยอมรับเทคโนโลยีตามกรอบ Technology Acceptance Model ของ {Davis, 1989 #2} อยู่ในระดับใด และมีแนวโน้มใช้งานระบบในระยะยาวมากน้อยเพียงใด
4. รูปแบบการคิดค่าบริการและโครงสร้างการตัดสินใจซื้อที่เหมาะสมสำหรับการนำระบบไปใช้งานในระดับหลักสูตรและคณะอื่นในมหาวิทยาลัยเป็นอย่างไร

## 1.3 วัตถุประสงค์ของการศึกษา

การศึกษานี้มีวัตถุประสงค์หลักห้าประการ ซึ่งสอดคล้องกับคำถามวิจัยในหัวข้อ 1.2 ได้แก่

1. เพื่อศึกษาและทำความเข้าใจปัญหาและความต้องการของเจ้าหน้าที่หลักสูตรและนิสิตในการเข้าถึงข้อมูลหลักสูตรในระดับบัณฑิตศึกษา
2. เพื่อออกแบบและพัฒนาระบบผู้ช่วยตอบคำถามอัตโนมัติในชื่อ VIRIYA ที่ใช้เทคนิค Retrieval-Augmented Generation ร่วมกับสถาปัตยกรรม Multi-tenant SaaS บนแพลตฟอร์ม LINE Official Account
3. เพื่อประเมินคุณภาพคำตอบของระบบและความพร้อมในการนำไปใช้งานจริงจากมุมมองของเจ้าหน้าที่หลักสูตร
4. เพื่อประเมินการยอมรับเทคโนโลยีของนิสิตตามกรอบ Technology Acceptance Model และศึกษาปัจจัยที่มีผลต่อความตั้งใจในการใช้งาน
5. เพื่อวิเคราะห์ความเป็นไปได้เชิงพาณิชย์ของการนำระบบไปใช้ในรูปแบบ B2B SaaS สำหรับสถาบันการศึกษาและศึกษารูปแบบการคิดค่าบริการที่เหมาะสม

## 1.4 ขอบเขตของการศึกษา

การศึกษานี้มีขอบเขตของการดำเนินการในห้ามิติ คือ มิติของกลุ่มตัวอย่าง มิติของเนื้อหาในระบบ มิติของภาษา มิติของช่องทางการให้บริการ และมิติของระยะเวลา

**มิติของกลุ่มตัวอย่าง** ผู้วิจัยเลือกหลักสูตรสหสาขาวิชาธุรกิจเทคโนโลยีและการจัดการนวัตกรรม (TIP) จุฬาลงกรณ์มหาวิทยาลัยเป็น tenant แรกสำหรับการพัฒนาและทดสอบระบบ และสัมภาษณ์เจ้าหน้าที่จากหลักสูตรสหสาขาวิชาการจัดการสารอันตรายและสิ่งแวดล้อม (HSM) และหลักสูตรสหสาขาวิชาวิทยาศาสตร์สิ่งแวดล้อม เพื่อศึกษาความเหมือนและความแตกต่างของบริบทการให้บริการข้อมูลหลักสูตรในระดับบัณฑิตศึกษา โดยหลักสูตรสองหลักหลังไม่ได้รับการนำระบบไปใช้งานจริงในการศึกษานี้ แต่เป็นข้อมูลอ้างอิงเพื่อออกแบบสถาปัตยกรรมให้รองรับการขยายผลในอนาคต

**มิติของเนื้อหาในระบบ** ฐานความรู้ของระบบครอบคลุมเอกสารของหลักสูตร TIP ที่ประกอบด้วย ตารางเรียน ระเบียบข้อบังคับการสอบโครงร่างวิทยานิพนธ์และการสอบจบ ประกาศเกณฑ์การประเมินวิทยานิพนธ์ ประกาศทุนการศึกษา และคู่มือกระบวนการทั่วไปของหลักสูตร โดยในช่วงการศึกษานี้มีเอกสารทั้งหมด 14 ไฟล์ แยกเป็น chunk ใน vector database จำนวน 148 ชุด

**มิติของภาษา** ระบบรองรับการถามและการตอบทั้งภาษาไทยและภาษาอังกฤษ โดยตัดสินใจเลือกภาษาของคำตอบจากคำถามของผู้ใช้งานผ่านการตรวจจับสัดส่วนตัวอักษรของแต่ละสคริปต์

**มิติของช่องทางการให้บริการ** ระบบเปิดให้ใช้งานผ่านช่องทาง LINE Official Account ของหลักสูตร TIP สำหรับนิสิต และผ่านเว็บเบราว์เซอร์ที่ Admin Portal สำหรับเจ้าหน้าที่หลักสูตร ไม่ครอบคลุมช่องทางอื่น เช่น อีเมล โทรศัพท์ หรือการประชุมทางไกล

**มิติของระยะเวลา** การศึกษาดำเนินการในช่วงเดือนมกราคมถึงเมษายน 2569 ประกอบด้วย 3 ขั้นตอนที่สืบเนื่องกันตามที่กล่าวในบทที่ 3

## 1.5 วิธีดำเนินการวิจัยโดยย่อ

การศึกษานี้ใช้แนวทางการวิจัยแบบผสมผสานเชิงคุณภาพและเชิงปริมาณ (Mixed Methods Research) ที่แบ่งกระบวนการออกเป็นสามขั้นตอนที่สืบเนื่องกัน โดยใช้ผลลัพธ์ของขั้นตอนก่อนเป็นข้อมูลนำเข้าสำหรับขั้นตอนถัดไป ดังนี้

**ขั้นตอนที่ 1 การศึกษาและทำความเข้าใจปัญหา (Exploration Phase)** เป็นการวิจัยเชิงคุณภาพผ่านการสัมภาษณ์เชิงลึกกับเจ้าหน้าที่หลักสูตรจำนวน 4 ท่านจากสามหลักสูตรระดับบัณฑิตศึกษา และนิสิตในหลักสูตร TIP จำนวน 3 ท่าน ใช้วิธีการสัมภาษณ์แบบกึ่งโครงสร้าง (Semi-Structured Interview) และวิเคราะห์ข้อมูลด้วยเทคนิค Thematic Analysis เพื่อสกัดประเด็นปัญหาและข้อกำหนดของระบบ

**ขั้นตอนที่ 2 การพัฒนาระบบ (Development Phase)** นำข้อกำหนดที่ได้จากขั้นตอนแรกมาออกแบบสถาปัตยกรรมและพัฒนาระบบ VIRIYA ในรูปแบบ Iterative Development ที่มีการทดสอบอัตโนมัติตามหลัก Test-Driven Development กำกับทุกขั้นตอน ระบบประกอบด้วยส่วนประกอบหลักสี่ส่วน ได้แก่ Chat API, Ingestion Worker, Admin API และ Admin Portal ที่ทำงานร่วมกันบนแพลตฟอร์ม Google Cloud Run

**ขั้นตอนที่ 3 การประเมินและทดสอบระบบ (Evaluation Phase)** ทดสอบระบบต้นแบบกับผู้ใช้งานจริงในสี่รูปแบบที่ครอบคลุมทั้งการวัดคุณภาพของคำตอบและการยอมรับเทคโนโลยี ได้แก่ การประเมินคุณภาพคำตอบของแชทบอทโดยเจ้าหน้าที่หลักสูตร การประเมินการใช้งานระบบจัดการหลังบ้านผ่านสถานการณ์จำลอง 3 งาน การสำรวจการยอมรับเทคโนโลยีของนิสิตตามกรอบ Technology Acceptance Model และการสัมภาษณ์เชิงลึกภายหลังการประเมินเพื่อเก็บข้อมูลเชิงธุรกิจ

รายละเอียดของเครื่องมือวิจัย กลุ่มตัวอย่าง และวิธีการวิเคราะห์ข้อมูลในแต่ละขั้นตอนปรากฏในบทที่ 3

## 1.6 ประโยชน์ที่คาดว่าจะได้รับ

ประโยชน์ที่คาดว่าจะได้รับจากการศึกษานี้แบ่งออกเป็นสามมิติ ดังนี้

**ประโยชน์เชิงวิชาการ** การศึกษาเป็นการประยุกต์ใช้เทคนิค Retrieval-Augmented Generation ในบริบทการศึกษาของมหาวิทยาลัยไทย ซึ่งเป็นพื้นที่ที่ยังมีการศึกษาเชิงประจักษ์จำกัด องค์ความรู้ที่ได้จะครอบคลุมประเด็นการออกแบบสถาปัตยกรรม Multi-tenant สำหรับเอกสารภาษาไทยที่มีโครงสร้างซับซ้อน ตลอดจนการวิวัฒนาการของสถาปัตยกรรมจากรูปแบบที่ต้องเขียนตรรกะแยกตามประเภทเอกสาร (รุ่น v1) ไปสู่รูปแบบที่ใช้ความสามารถของโมเดล multimodal เดียว (รุ่น v2) บทเรียนจากการพัฒนาสามารถใช้เป็นแนวทางสำหรับผู้พัฒนาระบบ RAG ในบริบทการศึกษาอื่นได้

**ประโยชน์เชิงปฏิบัติ** ระบบที่พัฒนาขึ้นสามารถลดภาระการตอบคำถามซ้ำของเจ้าหน้าที่หลักสูตรได้ตามที่ผู้ประเมินในการศึกษานี้ประมาณการไว้ในช่วง 14 ถึง 17 ชั่วโมงต่อสัปดาห์ ทำให้เจ้าหน้าที่มีเวลาสำหรับงานที่ต้องใช้ดุลพินิจและความละเอียดเพิ่มขึ้น ขณะที่นิสิตได้รับคำตอบที่รวดเร็วและเข้าถึงได้ตลอด 24 ชั่วโมงโดยไม่ต้องรอเวลาราชการ

**ประโยชน์เชิงพาณิชย์** ผลการศึกษาให้ข้อมูลเชิงลึกเกี่ยวกับโครงสร้างการตัดสินใจซื้อของสถาบันการศึกษา รูปแบบการคิดค่าบริการที่ผู้ใช้งานเห็นว่าเหมาะสม และความเป็นไปได้ในการขยายผลไปยังคณะและมหาวิทยาลัยอื่น ซึ่งสามารถนำไปพัฒนาเป็นผลิตภัณฑ์ B2B SaaS ที่ตอบโจทย์ตลาดการศึกษาไทยได้ในระยะถัดไป

## 1.7 นิยามศัพท์เฉพาะ

คำศัพท์เฉพาะที่ใช้ในการศึกษานี้มีนิยามดังต่อไปนี้

**Retrieval-Augmented Generation (RAG)** หมายถึง เทคนิคการสร้างคำตอบของโมเดลภาษาขนาดใหญ่ที่ผสานการสืบค้นเอกสารที่เกี่ยวข้องเข้ามาเป็นบริบทของคำถามก่อนส่งให้โมเดลสร้างคำตอบ ตามแนวคิดของ {Lewis, 2020 #1}

**โมเดลภาษาขนาดใหญ่ (Large Language Model: LLM)** หมายถึง โมเดลปัญญาประดิษฐ์ที่ฝึกบนข้อมูลข้อความจำนวนมากและมีความสามารถในการทำความเข้าใจและสร้างข้อความภาษาธรรมชาติได้ ในการศึกษานี้ใช้โมเดล Claude Opus 4.7 ของบริษัท Anthropic เป็นโมเดลหลัก และ Claude Haiku 4.5 เป็นโมเดลสำหรับงานย่อย

**Agentic RAG** หมายถึง การขยายแนวคิด RAG พื้นฐานโดยให้โมเดลภาษาขนาดใหญ่ทำหน้าที่เป็น agent ที่สามารถตัดสินใจเลือกใช้เครื่องมือ (tools) ในการสืบค้น ประเมินความเพียงพอของข้อมูล และเรียกใช้เครื่องมือเพิ่มเติมในรอบถัดไปได้ตามความจำเป็น ซึ่งแตกต่างจาก RAG แบบ single-pass ที่สืบค้นและสร้างคำตอบในรอบเดียว

**Multi-tenant Software-as-a-Service (SaaS)** หมายถึง สถาปัตยกรรมซอฟต์แวร์ที่ระบบเดียวให้บริการแก่องค์กรหลายแห่ง (tenants) พร้อมกันโดยที่ข้อมูลของแต่ละองค์กรถูกแยกจากกันและแชร์ทรัพยากรพื้นฐานเพื่อประหยัดต้นทุน ตามคำจำกัดความของ {Bezemer, 2010 #3}

**LINE Official Account (LINE OA)** หมายถึง บัญชี LINE ประเภทธุรกิจที่องค์กรสามารถใช้สื่อสารกับผู้ใช้งานผ่านข้อความอัตโนมัติและ webhook ที่เชื่อมต่อกับระบบหลังบ้านภายนอก

**Namespace** หมายถึง กลไกการแยกข้อมูลใน Pinecone vector database ที่แต่ละ namespace จัดเก็บเวกเตอร์แยกจาก namespace อื่น การสืบค้นจำกัดอยู่ภายใน namespace ที่ระบุเท่านั้น ระบบ VIRIYA ใช้ namespace หนึ่งต่อหนึ่งหลักสูตรเพื่อรับประกันการแยกข้อมูลระดับ tenant

**Chunk** หมายถึง ส่วนย่อยของเอกสารที่ถูกแบ่งออกมาเพื่อเก็บในรูปเวกเตอร์ แต่ละ chunk มีขนาดเหมาะสมสำหรับการสืบค้นและการใช้เป็นบริบทของคำถาม พร้อม metadata ที่บ่งบอกแหล่งที่มาของ chunk

**Embedding** หมายถึง การแปลงข้อความให้เป็นเวกเตอร์ตัวเลขในปริภูมิหลายมิติ (1,536 มิติในการศึกษานี้) ที่ข้อความที่มีความหมายใกล้เคียงกันจะมีเวกเตอร์ที่อยู่ใกล้กันในปริภูมินั้น ใช้สำหรับการสืบค้นเชิงความหมาย (semantic search)

**Technology Acceptance Model (TAM)** หมายถึง แบบจำลองเชิงทฤษฎีที่อธิบายปัจจัยที่มีผลต่อการยอมรับเทคโนโลยีของผู้ใช้งาน โดยระบุตัวแปรหลักสองประการคือ การรับรู้ประโยชน์ (Perceived Usefulness) และ การรับรู้ความง่ายในการใช้งาน (Perceived Ease of Use) ตามข้อเสนอของ {Davis, 1989 #2} การศึกษานี้ขยายกรอบ TAM ด้วยการเพิ่ม construct เรื่องความน่าเชื่อถือ (Credibility) และความตั้งใจในการใช้งาน (Intention to Use)
